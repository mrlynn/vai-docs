# Research: v1.31.0 Nano Docs

## Objective

Document the shipped `v1.31.0` `voyage-4-nano` local inference capability and normalize release messaging across the `vai-docs` site.

## Primary Source

- `vai-nano-local-inference-spec.md`

## Confirmed Product Decisions

- Current release to document: `v1.31.0`
- Nano is already implemented and should be presented as shipped capability
- The Python bridge is user-facing and should be documented explicitly
- Scope includes existing docs updates, new docs pages, release notes, homepage/banner/install changes, and site-wide release normalization

## Key Feature Narrative

The nano release changes the onboarding story from API-key-first to a local-first option:

- install `vai`
- run `vai nano setup`
- embed locally with `--local`
- optionally store/search with MongoDB
- move to API-backed models later without changing the broader Voyage 4 family story

The central product story is:

- zero API key required for first-use local inference
- shared embedding space with the rest of the Voyage 4 family
- lightweight Python bridge enables local model execution inside a Node.js CLI

## Site Architecture Findings

- The docs site has no Docusaurus versioning
- There is no `versions.json`
- There is no `versioned_docs/`
- Release messaging is surfaced through live content, banner text, and blog posts

## Known Current-State Mismatches

- `docusaurus.config.ts` announcement bar still promotes `v1.29.0`
- historical release-note content exists for older versions such as `v1.25.0`
- several docs still imply an API-key-first onboarding flow

## Existing Pages Likely Requiring Updates

- `docs/intro.md`
- `docs/getting-started/installation.md`
- `docs/models/voyage-4-family.md`
- `docs/commands/embeddings/embed.mdx`
- `docs/commands/rag-pipeline/pipeline.md`
- `docs/commands/tools-and-learning/explain.mdx`
- `docusaurus.config.ts`

## New Pages Likely Needed

- a guide dedicated to nano/local inference setup and usage
- a conceptual page explaining local inference, shared space, and the Python bridge

## Suggested Information Architecture

Add nano documentation in places where users naturally look:

- Getting Started or Guides for setup and first use
- Models for model family positioning
- Command docs for `embed`, `pipeline`, and `explain`
- Release blog for launch framing

## Risks

- Over-explaining the Python bridge could make the feature feel more complex than it is
- Under-explaining it would conflict with the explicit user requirement that it be user-facing
- Release normalization must distinguish between historical archive content and "current release" messaging

## Execution Recommendation

Start with high-visibility entry points and release surfaces, then update reference pages, then add deep-dive nano docs, then close with blog/review/build validation.
